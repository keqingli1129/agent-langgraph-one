quickstart.py — Line-by-line explanation

File summary:
- Purpose: Interactive quickstart and setup script to configure local development for the Databricks agent template. It checks prerequisites, sets up .env, authenticates Databricks CLI profiles, optionally binds to existing Databricks app resources, creates or reuses an MLflow experiment, provisions or configures Lakebase (autoscaling or provisioned), and updates `databricks.yml` and `.env` accordingly.
- Main functions: `_load_yml`, `_save_yml`, `check_prerequisites`, `check_missing_prerequisites`, `check_node_version`, `setup_env_file`, `update_env_file`, Databricks auth helpers, MLflow experiment creation, Lakebase setup and validation, `update_databricks_yml_lakebase`, and `main` orchestration.

Imports:
- argparse, json, os, platform, re, secrets, shutil, subprocess, sys, pathlib.Path — core utilities and CLI handling.
- ruamel.yaml — round-trip YAML reading/writing preserving comments/formatting.

Non-obvious behaviors / external dependencies:
- Requires `databricks` CLI, `uv` task runner, Node.js/npm for frontend tooling, and `ruamel.yaml` Python package.
- Interacts with Databricks REST APIs via the `databricks` CLI and uses the Databricks Python SDK (`WorkspaceClient`) when available.
- Updates `databricks.yml` carefully to insert or modify Lakebase environment variable blocks and resource blocks while preserving formatting and comments.
- Handles both autoscaling and provisioned Lakebase variants; can create autoscaling project + branch via the SDK.
- Many operations are interactive; script supports non-interactive flags to drive CI usage.

Questions / ambiguous points:
- Should updates to `databricks.yml` be validated with a YAML schema or backed up before modifications?
- Any additional Lakebase env keys or resource patterns to support for non-standard templates?
