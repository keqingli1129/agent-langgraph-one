grant_lakebase_permissions.py — Line-by-line explanation

File summary:
- Purpose: Grant Lakebase (Postgres) permissions to a Databricks Apps service principal so the agent's memory tables/sequences are accessible.
- Main functions: `_grant_permissions` (performs grants via LakebaseClient helpers) and `main` (parses CLI args, selects Lakebase instance/project/branch, creates role for SP, calls `_grant_permissions`).

Imports:
- `argparse` — parse CLI arguments.
- `os`, `sys` — environment variables and stderr/exit.
- `dotenv.load_dotenv` — load .env variables.
- `databricks_ai_bridge.lakebase` (used inside functions) — Lakebase client and privilege enums.
- `re` — regex used conditionally to parse autoscaling endpoint.

Non-obvious behaviors / external dependencies:
- Requires `databricks_ai_bridge.lakebase` package and Lakebase access.
- Reads LAKEBASE_* env vars from `.env` via `load_dotenv()`.
- Supports both provisioned Lakebase instances (`--instance-name`) and autoscaling endpoints (project+branch or `--autoscaling-endpoint`).
- Creates a role for the service principal (type "SERVICE_PRINCIPAL"); if role exists, it continues.
- Grants schema/table privileges and sequence privileges; many grants are guarded with try/except and will warn (not fail) if objects don't exist yet.
- Uses conventions: `is_mcp_connection` style in other scripts; here sequence/list schema names are configured in constants.
- Some behavior depends on database schema names and presence of tables; fresh branches may require re-running after first agent run.

Questions / ambiguous points:
- Should failures be surfaced more verbosely or logged to a file?
- Any additional schemas/tables that should be included for other memory types?
- Should role creation use a more specific role name or annotations?
