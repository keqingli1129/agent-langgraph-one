preflight.py — Line-by-line explanation

File summary:
- Purpose: Run a local pre-flight check by starting the local agent server, performing a /health check and a test /invocations request to verify the agent responds.
- Main functions: `find_free_port`, `start_server`, `stop_server`, `check_health`, `check_invocations`, and `main` which orchestrates the flow.

Imports:
- `json` — parse JSON responses.
- `os`, `socket`, `subprocess`, `sys`, `threading`, `time` — OS and process management, networking, concurrency, and timing utilities.
- `urllib.request`, `urllib.error` — make HTTP requests for health and invocations.

Non-obvious behaviors / external dependencies:
- Requires `uv` CLI task `start-server` to exist (`uv run start-server --port <port>`).
- Reads platform via `sys.platform` to set process-group handling for signals on POSIX vs Windows.
- Starts server in a subprocess and reads stderr lines in a background thread to detect startup messages.
- Waits up to `SERVER_START_TIMEOUT` seconds for startup and `REQUEST_TIMEOUT` for invocation responses.
- If server exits early, it prints the last stderr lines and exits with failure.
- Uses simple JSON payload for `/invocations` and checks for presence of non-empty `output` array.

Questions / ambiguous points:
- Should the script log more HTTP response details on failure? Currently prints a truncated JSON preview on unexpected shapes.
- Is `uv run start-server` guaranteed available in the developer environment?
